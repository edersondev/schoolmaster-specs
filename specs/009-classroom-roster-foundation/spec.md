# Feature Specification: Backend Classroom Roster Foundation

**Feature Branch**: `009-classroom-roster-foundation`  
**Created**: 2026-05-29  
**Status**: Ready for Implementation  
**Input**: User description: "Define the next SchoolMaster backend implementation slice after completed account lifecycle workflows. The backend must introduce classroom, course, section, group, roster membership, teacher assignment, and academic-period scoping foundations. Preserve contract-first delivery, tenant-by-column isolation with School as tenant root and school_id for school-owned records, explicit platform versus school authorization separation, API-only /api/v1 behavior, OpenAPI-aligned response envelopes, validation, lifecycle status handling, active academic-period rules, audit events, and migration impact on existing direct student assignments before implementation. Do not include teacher workflow correction behavior, guardian self-service, report lifecycle expansion, frontend implementation, billing, messaging, platform-wide support access, advanced assessment content types, or undocumented APIs in this slice."

## Clarifications

### Session 2026-05-29

- Q: Which teaching structure hierarchy is approved for v1? -> A: Use one primary ClassSection/Roster model with optional course, classroom, section, or group metadata instead of separate top-level resources.
- Q: How should existing direct learning-set student assignments behave after rosters are introduced? -> A: Existing direct assignments become read-only legacy records; new roster-aware workflow writes must use roster memberships once those workflows are approved.
- Q: Who may manage teacher assignments? -> A: Teacher assignment management is limited to authorized school administrators.
- Q: How should duplicate or overlapping roster memberships be handled? -> A: Reject overlapping active memberships for the same student, roster, and academic period.
- Q: Which lifecycle states apply to roster records? -> A: ClassSection/Roster uses active/inactive; RosterMembership uses active/ended; TeacherAssignment uses active/inactive.
- Q: How should duplicate active teacher assignments be handled? -> A: Reject duplicate active assignments for the same teacher, ClassSection/Roster, and academic period.
- Q: How should batch roster membership changes behave? -> A: Allow batch membership changes and reject the whole request if any member is invalid.
- Q: Which ClassSection/Roster identifier must be unique? -> A: `code` is unique per school and academic period; names may repeat.
- Q: How should ClassSection/Roster inactivation handle active memberships or teacher assignments? -> A: Reject inactivation while active memberships or teacher assignments exist.
- Q: What roster assignment visibility do teachers receive in this slice? -> A: Teachers may list and read only their own active teacher assignments.
- Q: What effective-date rules apply to roster lifecycle changes? -> A: Effective dates must fall within the academic period and be today or past dates only.
- Q: When are lifecycle reasons required? -> A: Reasons are required for membership ending, teacher assignment deactivation, and roster inactivation.
- Q: Should v1 expose teacher assignment detail retrieval? -> A: Include `GET /api/v1/teacher-assignments/{teacherAssignmentId}` for school administrators and for teachers reading only their own active assignment.
- Q: What is the v1 batch size limit for roster membership changes? -> A: Maximum 100 membership changes per batch request.
- Q: How should concurrent conflicting roster, membership, or teacher-assignment writes be handled? -> A: Resolve conflicts transactionally so one request succeeds and the other returns the documented conflict response without partial state.
- Q: How should effective start dates be handled for roster membership and teacher assignment creation? -> A: Require explicit effective start dates for roster membership creation and teacher assignment creation.
- Q: How should course, classroom, section, and group data be stored in v1? -> A: Store course/classroom/section/group as structured metadata fields on ClassSection/Roster; do not create separate internal Course/Classroom/Section/Group tables in v1.
- Q: Which fields are allowed in each structured course/classroom/section/group metadata block? -> A: Each metadata block may include optional `code` and `name` only.
- Q: Which timezone defines today for effective-date validation? -> A: Use the resolved school's configured timezone; if unavailable, use the application default timezone.
- Q: Which fields must roster foundation audit events include? -> A: Audit events must include actor user ID, school ID, target type/ID, action, outcome, lifecycle reason when present, and tenant-safe summary metadata.
- Q: Which list filters are approved for v1 roster foundation operations? -> A: Lists support pagination plus `academicPeriodId` and `status` filters only; no include expansion unless OpenAPI later adds it.
- Q: What pagination limits apply to v1 roster foundation list operations? -> A: List operations use default page size 25 and maximum page size 100.
- Q: Can inactive ClassSection/Roster records be reactivated in v1? -> A: Inactive ClassSection/Roster records cannot be reactivated in v1; administrators must create a new roster if needed.
- Q: What initial status is allowed when creating a ClassSection/Roster? -> A: ClassSection/Roster creation always creates `active` records; `inactive` is reached only through inactivation.
- Q: What student enrollment rule determines roster membership eligibility? -> A: Student must have active same-school enrollment covering the membership effective start date.
- Q: What teacher role rule determines teacher assignment eligibility? -> A: Teacher must have active same-school teacher-compatible role on the assignment effective start date.
- Q: What date-order rule applies when ending memberships or deactivating teacher assignments? -> A: Membership end and teacher deactivation effective dates must be on or after their effective start dates.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Define School Teaching Structures (Priority: P1)

A school administrator creates and maintains the school-owned class section or roster structures needed to organize students and teachers by academic period, with optional structured course, classroom, section, or group metadata fields on the ClassSection/Roster record. Each metadata block may include optional `code` and `name` only.

**Why this priority**: Teacher workflows currently depend on direct student selection. Group-based teaching needs explicit school-owned structures before future learning sets, grades, attendance, reports, or correction workflows can safely rely on rosters.

**Independent Test**: Authenticate as a school administrator with an active school context, create one class section or roster for an active academic period, retrieve it through the documented response envelope, and verify the same structure is invisible from another school context.

**Acceptance Scenarios**:

1. **Given** a school administrator has an active resolved school context and an active academic period, **When** they create a valid class section or roster with optional structured course, classroom, section, or group metadata fields according to the approved contract, **Then** an active record is created for that school and period only.
2. **Given** teaching structures exist in multiple schools, **When** the administrator lists or views structures, **Then** only records from the resolved school context are returned.
3. **Given** a create or update request references a missing, inactive, closed, cross-tenant, or unsupported academic period, **When** the request is submitted, **Then** the request is rejected before creating or changing school-owned structure state.
4. **Given** the request tries to create an unsupported metadata shape or duplicate same-school code within the same academic period, **When** the request is submitted, **Then** the request is rejected with the documented validation or conflict response.
5. **Given** a ClassSection/Roster has active memberships or active teacher assignments, **When** a school administrator attempts to inactivate it, **Then** the request is rejected with the documented conflict response and the roster remains active.
6. **Given** a school administrator inactivates an eligible ClassSection/Roster, **When** the request omits a lifecycle reason, **Then** the request is rejected before roster state changes.

---

### User Story 2 - Manage Roster Membership (Priority: P2)

A school administrator assigns eligible student profiles to approved rosters for an academic period, tracks membership status over time, and preserves history when students leave, transfer, or become inactive.

**Why this priority**: Roster membership is the operational bridge between student enrollment and teacher workflows. It must preserve tenant-safe student lifecycle rules before teacher-facing workflows can move from direct student assignment to roster-aware assignment.

**Independent Test**: Add active same-school student profiles to a roster in an active academic period, list roster members, remove or inactivate one membership with a reason and effective date, and verify historical membership remains available to authorized school administrators.

**Acceptance Scenarios**:

1. **Given** a roster exists in the resolved school and active academic period, **When** a school administrator adds one or more active same-school student profiles, **Then** the memberships are created without exposing or modifying another school's students.
2. **Given** an active roster membership already exists for the same student, roster, and academic period, **When** the administrator submits a duplicate or overlapping active membership, **Then** the request is rejected with the documented conflict response and the existing membership remains unchanged.
3. **Given** a student profile is inactive, transferred out, missing, deleted, cross-tenant, or lacks active same-school enrollment covering the membership effective start date, **When** membership is requested, **Then** no roster membership is created.
4. **Given** a batch roster membership change includes one invalid student profile, duplicate active membership, cross-tenant reference, or student without active same-school enrollment covering the membership effective start date, **When** the request is submitted, **Then** the entire batch is rejected without creating, ending, or changing any membership in that request.
5. **Given** a roster member leaves the roster, **When** an authorized administrator records the membership end state with required effective date and reason, **Then** current membership changes while historical membership remains retained.
6. **Given** a roster membership change uses an effective date outside the selected academic period, a future effective date, or an end effective date before the membership effective start date, **When** the request is submitted, **Then** the request is rejected before membership state changes.

### User Story 3 - Assign Teachers to Teaching Structures (Priority: P3)

An authorized school administrator assigns eligible teachers to school-owned teaching structures for a defined academic period so later teacher workflows can validate access through roster and assignment context.

**Why this priority**: Teacher assignment rules determine who may operate against roster-owned learning sets, grades, attendance, and content in future slices. The authorization boundary must be explicit before existing teacher workflows start depending on roster ownership.

**Independent Test**: As a school administrator, assign an active same-school teacher with an active teacher-compatible role on the assignment effective start date to an approved teaching structure for an active academic period, verify the assignment appears only in the resolved school, and verify inactive, cross-tenant, missing-role-on-effective-start-date, duplicate, non-admin, or closed-period assignments are rejected.

**Acceptance Scenarios**:

1. **Given** an eligible teacher user belongs to the resolved school and has an active teacher-compatible role on the assignment effective start date, **When** an authorized school administrator assigns the teacher to an approved teaching structure, **Then** the assignment is recorded for that school and academic period only.
2. **Given** a teacher has active teacher assignments in the resolved school context, **When** the teacher lists or reads assigned teaching structures through a documented operation, **Then** only that teacher's own active assignments are returned.
3. **Given** an active teacher assignment already exists for the same teacher, ClassSection/Roster, and academic period, **When** a school administrator submits a duplicate active assignment, **Then** the request is rejected with the documented conflict response and the existing assignment remains unchanged.
4. **Given** a teacher is inactive, deleted, lacks an active same-school teacher-compatible role on the assignment effective start date, belongs to another school, or the target structure is inactive or closed, **When** assignment is attempted, **Then** the request is rejected without partial assignment state.
5. **Given** a teacher assignment deactivation omits a lifecycle reason, **When** the request is submitted, **Then** the request is rejected before assignment state changes.
6. **Given** a teacher assignment change uses an effective date outside the selected academic period, a future effective date, or a deactivation effective date before the assignment effective start date, **When** the request is submitted, **Then** the request is rejected before assignment state changes.
7. **Given** a platform administrator attempts to assign teachers inside a school without explicit permitted school context and school-scoped authorization, **When** the request is submitted, **Then** access is rejected rather than treated as a platform override.

### Edge Cases

- How does the backend reject missing, inactive, mismatched, or unauthorized `X-School-Id` tenant context before ClassSection/Roster, teacher assignment, or membership data access?
- How does the backend enforce the v1 decision that ClassSection/Roster is the primary teaching structure while course, classroom, section, and group values remain structured metadata fields with optional `code` and `name` only rather than separate top-level lifecycle resources or separate internal tables?
- How does the backend identify duplicate ClassSection/Roster records? `code` is unique per school and academic period; names may repeat.
- How does the backend keep existing direct learning-set student assignments available as read-only legacy records while preventing new direct-assignment writes once roster-aware workflow writes are approved?
- How does the backend reject teacher assignment management by teachers, lead teachers, department managers, platform administrators without school context, or any actor other than an authorized school administrator?
- How does the backend limit teacher read visibility? Teachers may list and read only their own active teacher assignments and cannot view other teachers' assignments or manage assignments.
- What happens when an academic period is closed, inactive, missing, outside the school's academic year, or not compatible with a student's enrollment history?
- What happens when a roster membership request references a student whose same-school enrollment does not cover the membership effective start date? The request is rejected before membership state changes.
- What happens when a roster lifecycle request uses a future effective date or an effective date outside the selected academic period? The request is rejected before state changes; today-or-past evaluation uses the resolved school's configured timezone, falling back to the application default timezone only when the school has no configured timezone.
- What happens when a lifecycle-ending request omits a required reason? Membership ending, teacher assignment deactivation, and roster inactivation are rejected without a reason.
- What happens when roster membership creation or teacher assignment creation omits an effective start date? The request is rejected before state changes.
- What happens when membership ending or teacher assignment deactivation uses an effective date before the record's effective start date? The request is rejected before state changes.
- What happens when roster membership date ranges overlap, duplicate, or conflict with transfer, inactive, or deleted student profile states? Overlapping active memberships for the same student, roster, and academic period are rejected without changing existing membership history.
- How does the backend handle batch roster membership changes when at least one member is invalid? The entire batch is rejected without partial membership creation, ending, or status changes.
- What happens when a roster membership batch exceeds the v1 size limit? The request is rejected before any membership state changes.
- What happens when a teacher assignment request references a teacher whose same-school teacher-compatible role is not active on the assignment effective start date? The request is rejected before assignment state changes.
- What happens when a teacher assignment duplicates an existing active assignment for the same teacher, ClassSection/Roster, and academic period? The duplicate assignment is rejected without changing the existing assignment.
- What happens when concurrent requests attempt conflicting roster code, membership, teacher assignment, lifecycle, or dependency-sensitive writes? The backend resolves the conflict transactionally so one request succeeds and the other returns the documented conflict response without partial state.
- What happens when a ClassSection/Roster inactivation would orphan active memberships or teacher assignments? The inactivation is rejected until active memberships are ended and active teacher assignments are deactivated.
- What happens when a school administrator attempts to reactivate an inactive ClassSection/Roster? Reactivation is not supported in v1; the request is rejected and administrators must create a new roster if needed.
- What happens when a school administrator attempts to create a ClassSection/Roster as inactive? Creation as inactive is rejected; v1 records start active and can become inactive only through the documented inactivation workflow.
- How does the backend preserve historical membership, teacher assignment, and teaching structure records when students, teachers, academic periods, or schools become inactive?
- How does the backend avoid granting platform administrators implicit access to school-owned roster and assignment data?
- How does the backend prevent roster introduction from changing existing student self-view, reporting, teacher correction, guardian self-service, or report lifecycle behavior without separate specifications?
- How does the backend avoid implementing frontend screens, billing, messaging, platform support access, advanced assessment types, or undocumented API behavior in this slice?

## Architecture & Contract Impact *(mandatory)*

### Repository Impact

- **Backend repository impact**: Implement a backend-only classroom and roster foundation for documented ClassSection/Roster records, roster membership, school-administrator-managed teacher assignment, lifecycle status, academic-period scoping, request validation, policy authorization, service-layer transition rules, response resources, tenant context enforcement, audit events, and regression tests.
- **Frontend repository impact**: No frontend implementation is included in this feature. Future roster, class-section, and teacher-assignment screens must consume only the OpenAPI operations approved by this slice.
- **Specification or contract repository impact**: OpenAPI must be expanded before backend exposure because current completed slices do not publish ClassSection/Roster, roster membership, or teacher assignment management operations.
- **Delivery ownership and sequencing**: `schoolmaster-specs` defines the classroom and roster boundary first, `schoolmaster-backend` implements only approved operations next, and `schoolmaster-frontend` consumes the behavior only after backend verification remains contract-compliant.

### API Contract Impact

- **OpenAPI update required**: Yes. The active feature contract and aggregate contract must define operation IDs, paths, request schemas, response schemas, structured course/classroom/section/group metadata fields with optional `code` and `name` only, lifecycle transitions, teacher assignment visibility, roster membership behavior, academic-period constraints, audit-relevant state transitions, and allowed actor roles before backend implementation begins.
- **Versioned endpoints affected**: Proposed contract surface is limited to `/api/v1/class-sections`, `/api/v1/class-sections/{classSectionId}/memberships`, `/api/v1/teacher-assignments`, and `/api/v1/teacher-assignments/{teacherAssignmentId}` unless OpenAPI approves a different path set.
- **JSON response impact**: Responses must use documented success, paginated, validation, unauthorized, forbidden, tenant-mismatch, inactive-record, conflict, oversized batch rejection, all-or-nothing batch rejection, and not-found envelopes. No backend-local envelope, ad hoc error code, undocumented status code, undocumented field, undocumented filter, undocumented include expansion, or undocumented sort behavior is approved.
- **Authentication/authorization impact**: All operations require authenticated access, an active permitted school context, and documented school-scoped permissions. Teacher assignment management is limited to authorized school administrators. Teachers may list and read only their own active teacher assignments where OpenAPI documents that visibility. Platform administration does not imply school-owned roster, membership, or teacher assignment access without an explicit permitted school context and approved school authorization.
- **Compatibility impact**: This is additive against current backend behavior. Existing direct learning-set student assignments become read-only legacy records once roster-aware workflow writes are approved. Any other change to existing direct learning-set assignment, grade, attendance, student self-view, reporting, account lifecycle, or administration lifecycle behavior requires explicit contract revision and compatibility review.

### Data & Tenancy Impact

- **Tenant scoping impact**: ClassSection/Roster, roster membership, teacher assignment, and lifecycle history records are school-owned records governed by `school_id`. Course, classroom, section, and group values are structured metadata fields on the primary teaching structure; v1 MUST NOT create separate internal Course/Classroom/Section/Group tables or expose separate lifecycle resources unless a future specification promotes them.
- **Cross-tenant or platform access impact**: Cross-school student profiles, teacher users, academic periods, teacher content, learning sets, grades, attendance, guardians, reports, and private content remain isolated to their original school unless a later specification explicitly approves a cross-tenant workflow.
- **Soft delete impact**: Teaching structures, roster memberships, and teacher assignments must preserve operational and audit history. ClassSection/Roster records start as `active` and can become `inactive` only through the documented inactivation workflow; RosterMembership records use `active` and `ended`; TeacherAssignment records use `active` and `inactive`. Permanent purge, anonymization, merge, or restore behavior is not part of this slice unless OpenAPI and specifications are updated first.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The backend implementation slice MUST be limited to documented ClassSection/Roster, roster membership, school-administrator-managed teacher assignment, lifecycle status, and academic-period scoping operations unless OpenAPI is updated first.
- **FR-002**: Every operation in this slice MUST resolve an active permitted school context before validation that depends on school-owned structures, student profiles, teacher users, academic periods, authorization decisions, persistence, lifecycle transitions, audit events, or response shaping.
- **FR-003**: Requests with missing, mismatched, inactive, or unauthorized school context MUST fail before any class section, roster, membership, teacher assignment, student profile, teacher user, academic record, or report data is accessed.
- **FR-004**: ClassSection/Roster creation and update MUST create or modify records only inside the resolved school context and MUST validate documented names, required codes, optional structured course/classroom/section/group metadata blocks, academic-period references, lifecycle status, actor permissions, and code uniqueness per school and academic period before persistence; names may repeat. ClassSection/Roster creation MUST create `active` records only; requests to create inactive records MUST be rejected. Each structured course/classroom/section/group metadata block MAY include optional `code` and `name` only. V1 MUST NOT create separate internal Course/Classroom/Section/Group tables. Concurrent duplicate-code writes MUST be resolved transactionally so only one request succeeds and conflicting requests return the documented conflict response without partial state.
- **FR-005**: Teaching structure listing and detail retrieval MUST return only records visible in the resolved school context. List operations MUST support documented pagination with default page size 25 and maximum page size 100, plus `academicPeriodId` and `status` filters only. No include expansion, additional filters, or sort behavior is approved unless OpenAPI later documents it.
- **FR-006**: The terms `Roster`, `ClassSection`, and `ClassSection/Roster` MUST refer to the same primary v1 teaching-structure model in this slice. Backend implementation MUST NOT introduce a separate roster persistence model, separate roster lifecycle resource, or separate internal Course/Classroom/Section/Group table unless a future specification and OpenAPI update approve that split.
- **FR-007**: Roster membership creation MUST validate every referenced student profile as active, same-school, and backed by active same-school enrollment covering the membership effective start date before membership is created.
- **FR-008**: Roster membership changes MUST preserve historical membership state, effective dates, actor context, lifecycle reasons, and audit events while preventing inactive or transferred students from participating in new roster-based workflows where prohibited. Creating a roster membership MUST require an explicit effective start date, ending a roster membership MUST require a reason, and the shared lifecycle/effective-date rules in FR-012 MUST apply. Overlapping active memberships for the same student, roster, and academic period MUST be rejected without changing the existing membership.
- **FR-008a**: Batch roster membership changes MUST be all-or-nothing and MUST accept no more than 100 membership changes per request. If a batch exceeds 100 requested changes, or if any requested member is invalid, inactive, transferred, deleted, cross-tenant, duplicate, overlapping, or lacks active same-school enrollment covering the membership effective start date, the entire request MUST be rejected without creating, ending, or changing any membership in that batch. Concurrent conflicting membership writes MUST be resolved transactionally so only one request succeeds and conflicting requests return the documented conflict response without partial state.
- **FR-009**: Teacher assignment creation MUST require an authorized school administrator, an explicit effective start date, and validation that the assigned user is active, same-school, has an active same-school teacher-compatible role on the assignment effective start date, and is authorized for the target teaching structure before assignment state is created. Teacher-compatible role checks MUST use the existing school role/permission authority that marks users as eligible for teacher workflows; this slice MUST NOT introduce a separate teacher role taxonomy. Teacher assignment deactivation MUST require a reason, and the shared lifecycle/effective-date rules in FR-012 MUST apply. Duplicate active assignments for the same teacher, ClassSection/Roster, and academic period MUST be rejected without changing the existing assignment. Concurrent conflicting teacher-assignment writes MUST be resolved transactionally so only one request succeeds and conflicting requests return the documented conflict response without partial state.
- **FR-010**: Teacher assignment listing and detail retrieval MUST expose only assignments visible to the authenticated actor within the resolved school context and documented permission scope. School administrators may list and read same-school teacher assignments where documented; teachers may list assignments and retrieve `GET /api/v1/teacher-assignments/{teacherAssignmentId}` only for their own active assignments and MUST NOT view other teachers' assignments or manage assignments. Teacher assignment list operations MUST support documented pagination with default page size 25 and maximum page size 100, plus `academicPeriodId` and `status` filters only unless OpenAPI later documents a narrower role-specific filter set.
- **FR-011**: Academic-period scoping MUST prevent creation of active teaching structures, rosters, memberships, or teacher assignments against inactive, closed, missing, cross-tenant, or incompatible academic periods unless OpenAPI explicitly documents an exception. For this slice, a compatible academic period is same-school, active, not closed, has usable start and end dates, and contains the relevant lifecycle effective date; roster membership and teacher assignment compatibility also depends on same-school enrollment or teacher-compatible role coverage on the effective start date.
- **FR-012**: Lifecycle transitions for teaching structures, rosters, memberships, and teacher assignments MUST reject unsupported transitions, missing reasons where required, dependency conflicts, lifecycle-ending effective dates before the target record's effective start date, and attempts that would orphan active memberships or assignments. Roster inactivation, roster membership ending, and teacher assignment deactivation MUST require lifecycle reasons. Roster membership and teacher assignment creation effective start dates, plus lifecycle-ending effective dates, MUST fall within the selected academic period and MUST be today or past dates only using the resolved school's configured timezone, falling back to the application default timezone only when the school has no configured timezone. Membership end effective dates and teacher assignment deactivation effective dates MUST be on or after their target record's effective start date. ClassSection/Roster status is limited to `active` and `inactive`, RosterMembership status is limited to `active` and `ended`, and TeacherAssignment status is limited to `active` and `inactive`. ClassSection/Roster records MUST start as `active`; inactive ClassSection/Roster records MUST NOT be reactivated in v1, and administrators must create a new roster if needed. Concurrent lifecycle or dependency-sensitive writes MUST be resolved transactionally without partial state.
- **FR-012a**: ClassSection/Roster inactivation MUST require a reason and MUST be rejected while the roster has any active roster memberships or active teacher assignments; administrators MUST end memberships and deactivate teacher assignments before inactivating the roster.
- **FR-013**: Existing direct teacher workflow assignments MUST remain available as read-only legacy records, and new direct student-assignment writes MUST NOT be introduced once roster-aware workflow writes are approved; any migration, backfill, or removal of legacy reads requires a future specification and contract update.
- **FR-014**: Backend audit events MUST record creation, update, roster inactivation, membership add/end, teacher assignment add/deactivate, lifecycle reasons for roster inactivation, membership ending, and teacher assignment deactivation, blocked cross-tenant attempts, and conflict outcomes for records covered by this slice. Each audit event MUST include actor user ID, school ID, target type and target ID, action, outcome, lifecycle reason when present, and tenant-safe summary metadata without exposing private student, teacher, credential, full request payload, or unauthorized cross-tenant data.
- **FR-015**: Backend authorization MUST keep platform administration, school administration, teacher workflow access, student self-view, guardian access, reporting, and support access permissions separate. Teacher read visibility for the teacher's own active assignments MUST NOT grant roster membership management, teacher assignment management, student roster visibility, or broader school administration access.
- **FR-016**: Backend validation MUST reject undocumented request fields, unsupported filters, unsupported include expansion, unsupported sort values, unsupported page-size values above 100, invalid payload shapes, metadata block fields other than optional `code` and `name`, invalid metadata or structure references, duplicate ClassSection/Roster codes within the same school and academic period, roster membership batches with more than 100 requested changes, overlapping active memberships for the same student, roster, and academic period, duplicate active teacher assignments for the same teacher, ClassSection/Roster, and academic period, invalid lifecycle transitions, missing effective start dates for roster membership or teacher assignment creation, missing lifecycle reasons where required, future effective dates based on the resolved school timezone or application default fallback, effective dates outside the selected academic period, lifecycle-ending effective dates before the target record's effective start date, inactive references, duplicate references, and cross-tenant student, teacher, school, academic-period, roster, or structure references.
- **FR-017**: Backend responses MUST match the published OpenAPI success and error semantics declared for each approved operation. The backend MUST NOT emit undocumented status semantics unless OpenAPI first documents them on the relevant operation.
- **FR-018**: Backend implementation MUST NOT expose teacher workflow correction behavior, guardian self-service, report lifecycle expansion, report output changes, frontend behavior, billing, messaging, notification-center, platform-wide support access, advanced assessment content types, permanent purge, or undocumented API behavior in this slice.
- **FR-019**: Backend tests MUST cover successful flows, validation failures, authorization failures, tenant isolation failures, inactive school/user/student/teacher/period handling, duplicate prevention, metadata conflicts, roster membership conflicts, invalid effective dates, missing required lifecycle reasons, all-or-nothing batch membership rejection, transactional conflict handling for concurrent duplicate or dependency-sensitive writes, school-administrator-only teacher assignment conflicts, teacher own-active-assignment read visibility, lifecycle transition failures, roster inactivation dependency conflicts, read-only legacy direct-assignment compatibility, history preservation, audit event coverage, and response shape for every operation in this slice.

### Key Entities *(include if feature involves data)*

- **ClassSection/Roster**: Primary school-owned academic-period-scoped teaching structure with a required `code` unique per school and academic period, a repeatable name, and optional structured course, classroom, section, or group metadata blocks containing optional `code` and `name` only, without separate internal Course/Classroom/Section/Group tables or separate top-level lifecycle resources in v1; new records start as `active`, inactive records cannot be reactivated in v1, and `inactive` is reached only through the documented inactivation workflow.
- **RosterMembership**: School-owned relationship between a roster and a student profile, including `active` or `ended` status, effective dates, lifecycle reason when ended, eligibility context, history, and actor metadata; an end effective date must be on or after the membership effective start date.
- **TeacherAssignment**: School-owned relationship between a teacher-compatible user and an approved teaching structure or roster for an academic period; status is `active` or `inactive`, with a lifecycle reason required when deactivated, and a deactivation effective date must be on or after the assignment effective start date.
- **LegacyDirectAssignment**: Existing direct relationship between teacher workflow records and selected student profiles that remains readable for history but is not the approved model for new roster-aware workflow writes.
- **StudentProfile**: Existing school-owned learner profile whose active, transferred, inactive, deleted, and enrollment-history states determine roster membership eligibility; membership creation requires active same-school enrollment covering the membership effective start date.
- **AcademicPeriod**: Existing school-owned period that scopes teaching structures, rosters, memberships, and teacher assignments.
- **User**: Authenticated actor and teacher identity whose active status, school role, and permissions determine teacher assignment eligibility and management authority; teacher assignment creation requires an active same-school teacher-compatible role on the assignment effective start date.
- **AuditEvent**: Tenant-safe record of structure, roster, membership, assignment, lifecycle, authorization, and conflict outcomes, including actor user ID, school ID, target type and target ID, action, outcome, lifecycle reason when present, and tenant-safe summary metadata only.

## Assumptions

- `006-backend-student-enrollment` has established student profile lifecycle and enrollment history rules that roster membership must respect.
- `004-backend-teacher-workflows` currently supports direct selected-student learning-set assignments, grades, and attendance; this feature must document compatibility before changing those behaviors.
- Current OpenAPI contracts do not yet publish ClassSection/Roster, membership, or teacher assignment management operations; this feature must expand contracts before backend implementation.
- Course, classroom, section, and group values are v1 structured metadata blocks on the primary ClassSection/Roster model, with optional `code` and `name` only, not separate internal tables or top-level lifecycle resources.
- ClassSection/Roster `code` is required and unique per school and academic period; display names may repeat.
- Teaching structures, roster memberships, and teacher assignments are school-owned records governed by `school_id`.
- Teacher assignment management is limited to authorized school administrators.
- Existing direct learning-set student assignments remain readable as legacy records but should not receive new write behavior after roster-aware workflow writes are approved.
- The v1 feature is administrative and backend-only; frontend screens and teacher-facing roster workflow changes are deferred until contract-compliant backend behavior exists.
- Guardian self-service, teacher correction workflows, report lifecycle expansion, platform-wide support access, advanced assessment content types, billing, messaging, notifications, permanent purge, and undocumented behavior remain outside this slice.
