# AGENTS.md

## Repository Purpose

This repository is the source of truth for:
- product specifications
- business rules
- architecture decisions
- API contracts
- SaaS rules
- diagrams
- implementation constraints

Frontend and backend implementations must follow this repository.

## Mandatory Workflow

1. Read all relevant files before modifying anything
2. Specifications are the source of truth
3. Do not invent requirements
4. Do not implement undocumented APIs
5. Update documentation whenever behavior changes
6. Keep frontend/backend contracts synchronized
7. OpenAPI must always reflect the current behavior

## Directory Responsibilities

### /specs
Feature specifications generated by Spec Kit.

### /api
OpenAPI contracts shared by frontend and backend.

### /docs
Architecture and engineering documentation.

### /decisions
Architecture Decision Records (ADR).

## Constraints

- Never delete existing specifications without explicit instruction
- Never change business rules without updating the specs
- Never change API behavior without updating openapi.yaml
- Keep documentation concise and implementation-oriented

## Architecture

- Multi-tenant SaaS
- tenant-by-column strategy with `School` as the v1 tenant root and `school_id`
  as the concrete tenant column for school-owned records
- Laravel API backend
- Vue 3 SPA frontend
- API-first architecture
- Separate repositories for frontend and backend

## Expected Behavior

When implementing or modifying features:
- first update specifications
- then update plans/tasks if needed
- then update API contracts
- only then implementation repositories should change
